## 好问题拆解

### 我看到的断点

原始材料已经有完整清单：前端设计、前端开发、全栈开发、代码审查、网页应用测试、CI/CD、Lint/格式修复、技术文档更新、查找 Skill。

真正需要解决的问题不是“有没有 10 个 Skill”，而是：读者看完以后，是否知道自己今天应该先用哪一个，以及在哪个研发环节把它用起来。

当前清晰度：中

最大缺口：目标读者和发布渠道还未知，所以只能先做通用内容选题方案，不能直接锁定最终标题语气。

## 问题说明书

我要分析的问题：
如何把「研发场景十大热门 Skills 推荐」做成一篇让研发同学愿意阅读、收藏，并实际尝试使用 Skills 的内容？

现象：
飞书 Wiki 已整理 10 个研发相关 Skills，但当前内容更像资料清单，缺少读者的使用路径和场景优先级。

目标：
改进选题角度，确定标题、内容结构、推荐顺序和读者行动。

核心冲突：
Skill 推荐天然容易写成“工具百科”，但研发同学真正关心的是“我现在遇到的这个任务，能不能少花半小时”。

背景事实：

- `frontend-design`：用于生产级前端界面设计、美化和重塑。
- `cache-components`：用于 Next.js PPR 和 Cache Components 场景。
- `fullstack-developer`：用于 JS/TS 技术栈的全栈开发任务。
- `frontend-code-review`：用于前端代码审查，强调质量、性能和业务逻辑。
- `code-reviewer`：用于通用代码审查，覆盖本地改动和远程 PR。
- `wepapp-testing`：Wiki 原文如此书写，用于基于 Playwright 的网页应用测试。发布前需确认是否应写作 `webapp-testing`。
- `pr-creator`：用于规范化创建 PR，并执行提交前检查。
- `fix`：用于修复格式和 linting 问题。
- `update-docs`：用于根据代码变更更新技术文档。
- `find-skills`：用于发现、搜索和安装更多 Skills。

约束：

- 不夸大单个 Skill 能力。
- 不把内容写成纯工具列表。
- 最好突出“研发流程”而不是“十大热门”。
- 若用于对外发布，需要保留安装、尝试、关注服务号等行动入口。

反馈入口：

- 标题点击率。
- 收藏率。
- 评论里是否有人问“怎么安装”“哪个适合我”。
- 是否有人反馈自己实际用了其中一个 Skill。
- 如果在内部发布，可观察团队成员是否在 PR、代码审查、测试、文档更新中开始调用 Skills。

请 Agent 做：

1. 生成 2-3 个候选选题角度。
2. 用“读者是否立刻知道怎么用”来批评每个角度。
3. 选出最适合发布的角度。
4. 给出标题、开头、正文结构和行动入口。

## 候选解释与批评

### 候选角度 A：十大热门 Skills 推荐

机制：用“热门清单”降低进入门槛，读者会因为“我是不是漏了什么好工具”而点开。

可观察信号：标题容易被收藏，但评论可能停留在“哪里下载”“有没有链接”，不一定产生具体使用。

行动变化：正文必须补充“你遇到什么任务，就先用哪一个”，否则会变成目录。

### 候选角度 B：研发全流程 Agent Skills 地图

机制：把 10 个 Skills 按研发流程排列，从设计、开发、审查、测试、PR、修复、文档到查找新 Skill，让读者把它当成工作流而不是清单。

可观察信号：读者更容易在评论里说“我先试代码审查/测试/PR 创建”，因为每个 Skill 对应一个具体工作环节。

行动变化：正文结构应从“Skill 名称介绍”改成“研发任务 -> 推荐 Skill -> 适用场景 -> 第一次怎么用”。

### 候选角度 C：别只让 Agent 写代码

机制：利用读者已有认知偏差：很多人只把 Agent 当代码生成器，但这份清单证明 Agent 可以参与更完整的研发协作。

可观察信号：标题更有传播性，适合公众号、小红书、短视频开头；但需要避免写成情绪化观点文。

行动变化：开头先指出误区，再用 10 个 Skills 展示“Agent 可以接管的研发边角料”。

## 推荐选题

主选题：

> 别只让 Agent 写代码：研发全流程最值得上手的 10 个 Skills

这个角度比“十大热门”更强，因为它制造了一个清楚的认知冲突：很多研发同学以为 Agent 的价值在写代码，但实际更大的价值可能在设计、审查、测试、PR、修复和文档这些高频任务里。

备选标题：

1. 从设计到 PR：研发每天最值得交给 Agent 的 10 个 Skills
2. 第一次用 Agent Skills，研发同学先从这 10 个开始
3. 不只是写代码：10 个能嵌入研发流程的 Agent Skills
4. 研发提效别再只盯代码生成，这 10 个 Skills 更像工作流外挂
5. 前端、测试、PR、文档：研发场景最实用的 10 个 Agent Skills

## 正文结构建议

### 1. 开头：先打破误区

很多人第一次用 Agent，默认只让它写代码。

但在真实研发流程里，最耗时间的不只是写代码，而是设计改来改去、PR 描述写不清、测试场景漏掉、Lint/格式反复失败、文档永远跟不上代码变更。

所以这份清单不是“10 个工具推荐”，而是一张研发流程里的 Agent Skills 地图：你在哪个环节卡住，就先用对应的 Skill。

### 2. 用流程重排 10 个 Skills

| 研发环节 | 推荐 Skill | 读者心里的问题 | 内容表达重点 |
|---|---|---|---|
| 界面设计 | `frontend-design` | 页面能不能别那么 AI 味？ | 生产级 UI、独特风格、重塑界面 |
| Next.js 前端开发 | `cache-components` | 缓存和 PPR 怎么不踩坑？ | 适合 Next.js Cache Components 场景 |
| 全栈开发 | `fullstack-developer` | 我能不能从页面到 API 一起做？ | JS/TS、React/Next.js、Node、数据库 |
| 前端审查 | `frontend-code-review` | TSX/JS 代码有没有性能和业务风险？ | 前端专项审查、结构化修复建议 |
| 通用审查 | `code-reviewer` | 合并前有没有隐藏问题？ | 本地改动和远程 PR 都能审 |
| 网页测试 | `webapp-testing` | 页面功能到底有没有跑通？ | Playwright、截图、控制台日志、交互验证 |
| PR 创建 | `pr-creator` | PR 描述和检查流程能不能规范？ | 模板、preflight、标题和描述 |
| 格式修复 | `fix` | 为什么 CI 又因为格式挂了？ | Prettier、linting、提交前清理 |
| 文档更新 | `update-docs` | 代码变了，文档怎么同步？ | 从代码 diff 映射到文档更新 |
| 查找新 Skill | `find-skills` | 我不知道有没有更合适的 Skill | 搜索、安装、扩展 Agent 能力 |

备注：Wiki 原文中“网页应用测试”名称写作 `wepapp-testing`，这里先按常见 Skill 名 `webapp-testing` 表达，发布前需要确认最终名称。

### 3. 推荐阅读路径

如果你是前端开发，先看：

- `frontend-design`
- `cache-components`
- `frontend-code-review`
- `webapp-testing`

如果你经常发 PR，先看：

- `code-reviewer`
- `pr-creator`
- `fix`
- `update-docs`

如果你刚开始用 Skills，先看：

- `find-skills`
- `fullstack-developer`
- `code-reviewer`

### 4. 结尾行动入口

可以用这个结尾：

> 第一次不用全部安装。
> 先选一个你今天最痛的研发环节：页面不好看、PR 不规范、CI 老失败、文档没人更新。
> 然后只试一个对应的 Skill。
> 只要它能帮你少掉一次返工，Skills 的价值就开始显现了。

## 推荐发布版本

### 标题

别只让 Agent 写代码：研发全流程最值得上手的 10 个 Skills

### 摘要

这不是一份单纯的工具清单，而是一张研发流程里的 Agent Skills 地图。从前端设计、Next.js 开发、全栈实现，到代码审查、网页测试、PR 创建、Lint 修复和文档更新，每个 Skill 都对应一个真实研发任务。你不需要一次性全部安装，先从今天最耗时间的那个环节开始。

### 文章骨架

1. Agent 不只适合写代码。
2. 真实研发流程里还有大量高频、重复、容易遗漏的任务。
3. 用“研发环节 -> 推荐 Skill”的方式介绍 10 个 Skills。
4. 给不同读者 3 条上手路径：前端、PR 高频用户、新手。
5. 用一个具体行动收尾：今天只试一个 Skill。
