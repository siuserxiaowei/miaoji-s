# 第一次用 Agent Skills，先收藏这 10 个

我一开始看 Agent Skills，也有点烦。

Agent 已经够多了，现在又冒出来一个 Skills。听起来像是又要学一套新概念。

但后来我发现，它其实没那么玄。

你可以把 Skill 当成一张小抄。Agent 本来就能干活，但每次干活之前，你都要告诉它：这件事按什么标准做，哪些坑不要踩，做完以后怎么检查。

Skill 就是把这些话提前写好。

所以它不太像“插件市场”里那种你装了以后就会多一个按钮的东西。它更像是给 Agent 加了一套工作习惯。

这也是为什么我觉得小白第一次看 Skills，不要先研究一堆名词。你就问自己一句：

我现在卡住的这件事，有没有对应的 Skill？

如果有，先试一个。别一次装十几个。

下面这 10 个，可以先收藏。你未必都会用上，但新手常见的几类问题，基本都能在里面找到对应入口。

## 先从“找 Skill”开始

如果你完全不知道该用哪个，先记住 `find-skills`。

这个 Skill 的作用很朴素：帮你找 Skill。

听起来有点绕，但对小白反而最实用。因为刚开始你不可能记得住那么多名字。你真正会说出口的问题通常是：

“有没有适合写技术文档的 Skill？”

“有没有适合 React 性能优化的 Skill？”

“我想做代码审查，应该找哪个？”

这时候就可以让 `find-skills` 先帮你找一轮。

我会建议新手把它当成入口，而不是把所有 Skill 名称背下来。会找，比记住更重要。

## 如果你在做页面，先看这两个

第一个是 `frontend-design`。

这个 Skill 适合解决一个很常见的问题：页面能跑，但不好看。

尤其是 AI 生成的页面，经常会有一种模板感。不一定错，只是你看一眼会觉得它没什么设计判断。卡片、按钮、标题都有，但放在一起就是很平。

这时候你可以让 Agent 用 `frontend-design` 重新处理界面。比如让它把一个普通首页改得更像正式产品，或者让它检查页面的排版、颜色、组件层级有没有问题。

它对不会设计的人挺友好，因为你不需要先讲一堆设计术语。你只要描述你不满意的地方，比如“太像模板”“没有重点”“不像一个要上线的页面”。

另一个是 `webapp-testing`。

很多人做完网页以后，验证方式就是自己点两下。

点得通，就觉得差不多了。

但网页的问题经常不在“看起来能不能打开”，而在流程里。登录能不能走完，表单能不能提交，控制台有没有报错，移动端布局有没有炸。

`webapp-testing` 可以让 Agent 用浏览器去测。它背后用的是 Playwright，可以做点击、输入、截图、看控制台日志这些事情。

如果你做了一个登录页，就别只让 Agent 看代码。直接让它跑一遍登录流程，看看从输入账号密码到进入首页，中间有没有问题。

这比“帮我检查一下代码”更具体，也更容易得到有用结果。

## 如果你想做一个完整小项目

这里可以看 `fullstack-developer`。

它适合那种“我不只是想做页面，我想把一个小产品跑起来”的场景。

比如你想做一个任务管理工具。页面是一部分，后面还会有登录、数据库、接口、权限。小白最容易卡在这里：每一块好像都懂一点，但不知道先后顺序。

`fullstack-developer` 可以帮你把这个想法拆开。前端有哪些页面，后端有哪些 API，数据库大概怎么设计，第一版先做什么。

它不能替你省掉所有学习成本，但它能帮你把混乱的东西排一下队。

还有一个更偏具体技术的 `cache-components`。

这个不一定适合所有小白。它主要面向 Next.js 项目里的缓存和 PPR 场景。你如果还没写到 Next.js，可以先跳过。

但如果你已经在做 Next.js，又经常听到 `use cache`、动态渲染、缓存失效这些词，它就有用了。它能帮你判断哪些数据适合缓存，哪些内容应该动态加载，数据更新后应该怎么处理缓存。

这个 Skill 不用硬啃。遇到 Next.js 性能和缓存问题时，再想起它就行。

## 写完代码以后，别急着提交

这里有三个 Skill 可以放在一起看：`frontend-code-review`、`code-reviewer`、`fix`。

`frontend-code-review` 更偏前端。

如果你改的是 `.tsx`、`.ts`、`.js` 这些文件，可以让它重点看前端代码的问题。比如有没有性能隐患，组件状态有没有乱用，某些业务场景会不会出 Bug。

`code-reviewer` 更通用。

它适合在提交前帮你扫一遍本地改动，也可以看远程 PR。你可以让它按严重程度列问题，而不是泛泛评价“代码质量不错”。

这点很重要。

新手用 AI 审代码，最怕得到一堆礼貌话。你要让它直接找问题，尤其是会影响功能、测试、安全、维护成本的问题。

然后是 `fix`。

这个名字很短，也很好理解。它主要处理格式和 linting 问题。

很多 CI 失败跟功能没关系。格式不对、变量没用、lint 规则没过，都可能让你在这种小事上耗半天。

如果你每次提交前都被这些小问题绊一下，可以先让 `fix` 跑一遍。

这三个 Skill 的顺序可以很简单：

先用 `fix` 清掉格式问题。
再用 `code-reviewer` 看整体风险。
如果改的是前端，再补一个 `frontend-code-review`。

这不是标准流程，只是比较省心。

## 到了协作环节，再看 PR 和文档

代码写完以后，还有两件事经常被低估。

一个是 PR。

很多人写代码没问题，但 PR 描述写得很随意。别人点进去不知道你改了什么、为什么改、测了没有。最后 Review 来回问，时间就耗在这种地方。

`pr-creator` 就是处理这个环节的。

它会帮你检查分支、找 PR 模板、跑一些提交前检查，然后生成更规范的标题和描述。

如果你在团队里协作，这个 Skill 会比你想象中实用。它不酷，但能少很多沟通成本。

另一个是文档。

对应的是 `update-docs`。

文档这件事很尴尬。大家都知道它重要，但真的改代码时，很容易忘。

新增一个参数，文档没写。
改了一个行为，示例还是旧的。
接口已经变了，说明页还停在上个版本。

时间一长，文档就没人敢信。

`update-docs` 做的事，就是根据代码变更提醒你哪些文档可能要更新。对于长期项目，这个比“写一篇漂亮文档”更重要。

漂亮不漂亮是一回事，跟不跟得上才是另一回事。

## 所以小白该怎么选？

别从 10 个里纠结。

我会这样选：

如果你还不知道 Skills 能干嘛，先用 `find-skills`。

如果你在做页面，先用 `frontend-design` 和 `webapp-testing`。

如果你想做完整项目，用 `fullstack-developer`。做 Next.js 缓存问题时，再看 `cache-components`。

如果你已经开始提交代码，用 `fix`、`code-reviewer`，前端项目再加 `frontend-code-review`。

如果你要发 PR 或维护文档，用 `pr-creator` 和 `update-docs`。

第一次不用全装。

真的，先用一个就行。

今天页面不好看，就用 `frontend-design`。
今天 PR 写不清，就用 `pr-creator`。
今天担心页面流程没跑通，就用 `webapp-testing`。

你只要在一个具体任务里用成一次，就会明白 Skills 的价值。

对我来说，Skills 的价值没那么玄：它会让 Agent 少一点临场发挥，多一点固定流程。

对小白来说，这已经够了。
